## Context

The auth service needs a stateless authentication mechanism that scales horizontally without shared session stores. The platform runs across multiple availability zones with auto-scaling enabled. Any solution that requires sticky sessions or a centralised session store introduces a single point of failure and complicates horizontal scaling.

## Decision

Use signed JWTs for all service-to-service and client-to-gateway authentication. No server-side session state. Tokens are signed with RS256 and validated locally by each service using the public key. Access tokens expire after 15 minutes. Refresh tokens are rotated on each use and stored only on the client.

## Consequences

All services must validate JWT tokens locally using the shared public key. Token revocation before expiry requires a deny list (Redis-backed, TTL-matched to token expiry). Refresh tokens must be rotated on every use to limit replay windows. The auth service is responsible for key rotation and publishing the JWKS endpoint.

## Rejected Alternatives

Session cookies were considered and rejected. They require a centralised session store (Redis or database-backed), which adds a single point of failure and conflicts with the horizontal scaling requirement. Session cookies also conflict with GDPR data minimisation requirements — storing session state server-side means retaining user-linked data beyond what is strictly necessary. Sticky sessions were ruled out as they are incompatible with auto-scaling across availability zones.
